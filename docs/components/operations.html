Operações

No contexto das operações do sistema, o Jenkins foi uma ferramenta fundamental no processo de desenvolvimento, uma vez que permitir uma integração constante e permite também fazer o deployment automatizado para um ambiente de produção.

Modos de execução

Para um melhor controlo, cada um destes estágios é opcional, isto é, é possível executar o pipeline com apenas alguns ou com todos. Para dar este tipo de liberdade, o pipeline esp23_OnTrack é um pipeline parametrizável, que podem ser escolhidos pelo utilizador no momento do build, para que este execute os passos pretendidos. Os diferentes modos de execução são: Build and Test, Test and Deploy, Deploy, Test, Build, Fast Deploy ou All.

Modo Explicação
Build and Test
Test and Deploy
Deploy Executa apenas as etapas relacionadas à implantação do pipeline, como implantar no Artifactoy, implantar o back-end, implantar a aplicação OnTrack e implantar no tempo de execução. Permite implantar a aplicação enquanto se saltam todos os testes.
Test Executa apenas as etapas relacionadas ao teste do pipeline, como teste, testes de Cucumber. Permite executar os testes de integração rapidamente, sem implantar a aplicação.

All Executa o pipeline completo.

Requisitos

A nível de requisitos para o deploy do sistema o que é necessário é uma máquina com a plataforma docker instalada, com um container a correr Kafka (porta 9092), um container a correr Zookeeper (porta 2181) e outro container a correr React (porta 3001). Assume-se ainda que nesta máquina o sistema de ELK com logstash está configurado através de um docker compose para ser possível fazer a configuração da monitorização.

Relativamente aos requisitos de hardware da máquina, estes são bastante básicos. A nível de RAM apenas será necessário cerca de 1GB, no entanto, a nível de memória em disco, dependendo do tempo de execução do sistema, o espaço ocupado poderá escalar, dado o armazenamento de uma grande quantidade de dados.

Para fazer o deploy apenas é necessário fazer pull das imagens do docker registry, criar os containers necessários para essas imagens e dar start aos mesmos, que no fundo, foi o que foi explicado na execução do Jenkinsfile. Para ter uma monitorização do mesmo, é necessário alterar o IP e porta do kafka para que o logback appender envie a informação recolhida para um tópico lido pelo logstash. Por sua vez, nos diferentes serviços é preciso especificar no código qual é esta porta e os dados são adicionados ao Elastic Search, onde podem depois ser analisados no kibana.

Neste sistema foi utilizado kubernetes e docker-compose, existindo então uma configuração out-of-the-box para o utilizador final, para além de se poder utilizar o jenkins.

Um dos aspetos menos positivos, foi a não utilização de environment variables, nem no projeto nem nos Jenkinsfiles, logo caso fosse necessário estender o sistema, as portas e IPs tinham de se alterar nos respetivos lugares. No entanto, como o sistema foi desenvolvido usando a plataforma Spring Boot, este aspeto não é muito preocupante, na medida em que as mesmas podem ser alteradas, facilmente, no ficheiro application.properties.

Monitorização

A monitorização é feita usando uma stack ELK (Elasticsearch Logstash Kibana).

Para fazer upload de nossos dados para o Elastic Search, utilizou-se um appender para logback. Este appender é configurado no logback.xml, disponível no código-fonte.

Esses dados podem então ser visualizados no Kibana, consultando o índice “esp30-logs”. Como todas as mensagens de log são carregadas lá, esses dados podem ser filtrados para mostrar todos os tipos de informações sobre o sistema em execução.
